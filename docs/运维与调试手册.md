## 一、CI/CD 自动部署（GitHub Actions + Self-hosted Runner）

### 架构

```
本地 push → GitHub Actions 触发 → 服务器 Runner 捡起任务 → git pull → 构建 → 重启服务
```

### 初次安装 Runner（新服务器）

```bash
# ===================== 后端 Runner =====================

mkdir -p /opt/actions-runner && cd /opt/actions-runner

# 下载 runner（如果墙，本地下载后 SCP 上传）
curl -o actions-runner-linux-x64.tar.gz -L \
  https://github.com/actions/runner/releases/download/v2.322.0/actions-runner-linux-x64-2.322.0.tar.gz
tar xzf actions-runner-linux-x64.tar.gz
echo "." > .auto_update    # 禁用自动更新（墙）

export RUNNER_ALLOW_RUNASROOT=1   # root 用户需要

# 去 GitHub 仓库 Settings → Actions → Runners → New self-hosted runner 拿 token
./config.sh --url https://github.com/Swing-G/LingXi --token XXXX --name backend-server --labels backend

sudo ./svc.sh install
sudo ./svc.sh start

# ===================== 前端 Runner =====================

mkdir -p /opt/actions-runner-frontend && cd /opt/actions-runner-frontend
cp /opt/actions-runner/actions-runner-linux-x64-2.322.0.tar.gz .
tar xzf actions-runner-linux-x64-2.322.0.tar.gz
echo "." > .auto_update

export RUNNER_ALLOW_RUNASROOT=1

# 去 https://github.com/Swing-G/LingXi-UI → Settings → Actions → Runners 拿 token
./config.sh --url https://github.com/Swing-G/LingXi-UI --token XXXX --name frontend-server --labels frontend

sudo ./svc.sh install
sudo ./svc.sh start
```

### Runner 重新注册（改名字/标签/换 token）

```bash
cd /opt/actions-runner    # 或 /opt/actions-runner-frontend

sudo ./svc.sh stop
sudo ./svc.sh uninstall

export RUNNER_ALLOW_RUNASROOT=1
./config.sh remove --token XXXX
./config.sh --url https://github.com/Swing-G/LingXi --token XXXX --name 新名字 --labels 新标签

sudo ./svc.sh install
sudo ./svc.sh start
```

### 项目目录结构（服务器）

```
/opt/
├── backend/                          # 后端仓库 clone
├── frontend/                          # 前端仓库 clone
├── application.yml                    # 后端配置文件
├── app.log                            # 后端运行日志
├── actions-runner/                    # 后端 Runner
├── actions-runner-frontend/           # 前端 Runner
└── deploy.log                         # 旧部署日志（可选）
```

## 二、systemd 服务配置

### 后端服务

```bash
sudo tee /etc/systemd/system/zhiguang.service << 'EOF'
[Unit]
Description=ZhiGuang Backend
After=network.target

[Service]
Type=simple
User=root
WorkingDirectory=/opt/backend
ExecStart=/usr/bin/java -Xms256m -Xmx384m -XX:+UseG1GC \
  -jar /opt/backend/target/zhiguang-1.0-SNAPSHOT.jar \
  --spring.config.location=/opt/application.yml
Restart=always
RestartSec=10
StandardOutput=append:/opt/app.log
StandardError=append:/opt/app.log

[Install]
WantedBy=multi-user.target
EOF

sudo systemctl daemon-reload
sudo systemctl enable zhiguang
sudo systemctl start zhiguang
```

### 常用命令

```bash
sudo systemctl status zhiguang          # 应用状态
sudo systemctl restart zhiguang         # 手动重启
sudo systemctl stop zhiguang            # 停止
tail -f /opt/app.log                    # 实时日志

sudo systemctl status actions.runner...  # Runner 状态
```

## 三、Nginx 配置

```bash
sudo tee /etc/nginx/sites-available/zhiguang << 'NGINX'
server {
    listen 80;
    server_name _;
    root /var/www/zhiguang;
    index index.html;

    location / {
        try_files $uri /index.html;    # SPA 路由回退
    }

    location /api/ {
        proxy_pass http://localhost:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 300s;        # SSE 流式长连接
        proxy_buffering off;            # 关闭缓冲（SSE 必须）
        proxy_cache off;
        proxy_set_header X-Accel-Buffering no;
    }
}
NGINX

sudo ln -sf /etc/nginx/sites-available/zhiguang /etc/nginx/sites-enabled/
sudo rm -f /etc/nginx/sites-enabled/default
sudo nginx -t && sudo systemctl reload nginx
```

## 四、Git SSH 配置（免密码 pull）

```bash
# 生成密钥
ssh-keygen -t ed25519 -C "server-deploy" -f ~/.ssh/github_deploy -N ""

# 显示公钥 → 复制添加到 https://github.com/settings/keys
cat ~/.ssh/github_deploy.pub

# 配置 SSH
cat > ~/.ssh/config << 'EOF'
Host github.com
    HostName github.com
    User git
    IdentityFile ~/.ssh/github_deploy
EOF
chmod 600 ~/.ssh/config

# 测试
ssh -T git@github.com   # 应输出 Hi Swing-G!

# 仓库改为 SSH 地址
cd /opt/backend  && git remote set-url origin git@github.com:Swing-G/LingXi.git
cd /opt/frontend && git remote set-url origin git@github.com:Swing-G/LingXi-UI.git
```

### 国内服务器墙的问题

- **下载文件**：本地下载 → SCP 上传
- **github.com 连不上**：配代理或换镜像
- **Runner 自动更新**：`echo "." > .auto_update` 禁用
- **git clone 慢**：改用 SSH 协议 + 密钥认证

## 五、部署 Workflow 文件

### 后端 `.github/workflows/deploy.yml`

```yaml
name: Deploy Backend

on:
  push:
    branches: [master]

jobs:
  deploy:
    runs-on: backend
    steps:
      - name: Pull latest code
        run: |
          cd /opt/backend
          git pull origin master

      - name: Build
        run: |
          cd /opt/backend
          mvn clean package -DskipTests -q

      - name: Restart App
        run: |
          sudo systemctl restart zhiguang
          sleep 5
          sudo systemctl is-active --quiet zhiguang && echo "✔ Deploy success" || echo "✘ Deploy failed"
          tail -5 /opt/app.log
```

### 前端 `.github/workflows/deploy.yml`

```yaml
name: Deploy Frontend

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: frontend
    steps:
      - name: Pull latest code
        run: |
          cd /opt/frontend
          git pull origin main

      - name: Install & Build
        run: |
          cd /opt/frontend
          npm install --legacy-peer-deps
          npm run build

      - name: Deploy to Nginx
        run: |
          sudo rm -rf /var/www/zhiguang/*
          sudo cp -r /opt/frontend/dist/* /var/www/zhiguang/
          sudo systemctl reload nginx
          echo "✔ Frontend deployed"
```

### ⚠️ 注意

- 后端分支是 `master`，前端分支是 `main`
- `runs-on` 标签要和 Runner 注册时的 `--labels` 一致
- `sudo` 命令不需要密码（root 用户）

## 六、RAG 问答 Bug 修复记录

### 问题现象

文章 RAG 问答返回「未找到相关内容，可能文章正文暂不可读」，日志显示 `contextsFound=0`。

### 根因 1：向量检索未按 postId 过滤

**文件**：`RagQueryService.java` — `searchContexts` 方法

**原因**：`vectorStore.similaritySearch()` 在全库范围搜 Top-K，然后在 Java 侧按 postId 后置过滤。用户问"总结"这类通用词时，其他文章切片相似度更高，目标文章切片被挤出 Top-K。

**修复**：
```java
// 修复后：把 postId 过滤推到 ES 向量检索层
List<Document> docs = vectorStore.similaritySearch(
    SearchRequest.builder()
        .query(query)
        .topK(topK)
        .filterExpression("postId == \"" + postId + "\"")  // ← 关键
        .build()
);
```

> ⚠️ 注意：ElasticsearchVectorStore 的 FilterExpressionConverter 会自动给 key 加 `metadata.` 前缀，所以 filter 里写 `postId` 而不是 `metadata.postId`。

### 根因 2：每次查询强制重建索引

**文件**：`RagIndexService.java` — `ensureIndexed` 方法

**原因**：`ensureIndexed` 调用 `reindexSinglePost(postId, true)`，每次 RAG 查询都重新下载 OSS→切片→调 embedding API，浪费且失败率高。

**修复**：改为 `reindexSinglePost(postId, false)`，利用 SHA256/ETag 指纹检测，仅在内容变化时重建。

### 根因 3：Embedding API 批大小超限

**文件**：`RagIndexService.java` — `reindexSinglePost` 方法

**原因**：OpenAI `text-embedding-v4` 限制每批最多 10 条。长文章切片 >10 个时 `vectorStore.add()` 返回 400 错误。更致命的是 `deleteExistingChunks` 先执行（先删旧后写新），add 失败后文章在向量库中变成 0 条切片。

**修复**：按每批 10 条分批调用 `vectorStore.add()`，单批失败不影响其他批次。
```java
int batchSize = 10;
for (int batchStart = 0; batchStart < chunks.size(); batchStart += batchSize) {
    List<Document> batchDocs = // ... 取这一批
    try {
        vectorStore.add(batchDocs);
    } catch (Exception e) {
        log.error("batch [{}-{}) failed", batchStart, batchEnd);
        // 继续下一批，不丢失已写入的数据
    }
}
```

### 面试话术

> "项目中有个 RAG 问答功能，用户对某些文章提问总是返回'未找到内容'。排查发现三个问题层层递进：一是向量检索没有限定文章 ID，全库搜索导致目标切片被挤出 Top-K；二是每次查询都 force 重建索引，浪费资源；三是 OpenAI embedding API 限制每批 10 条，长文章切片超限导致写入失败，加上代码先删后写的逻辑，失败后文章变成 0 条切片。修复后通过 filterExpression 下推过滤、指纹检测跳过重复索引、分批次写入解决。"

## 七、缓存一致性问题修复

### 问题现象

文章在数据库中改为 `private`，但前端 Feed 流仍然展示。

### 根因

**文件**：`KnowPostServiceImpl.java` — `invalidateCache` 方法

SQL 查询正确（`WHERE visible = 'public'`），详情页也有权限校验。但 `updateVisibility` 变更可见性时，只清理了**详情页缓存**，忘了清理 **Feed 流缓存**：

```
失效的：knowpost:detail:{id}:v1            ← 详情页
遗漏的：feed:item:{id}                      ← Feed 片段
遗漏的：feed:public:ids:*                   ← Feed ID 列表
遗漏的：Caffeine feedPublicCache            ← 本地页面缓存
```

前端请求 Feed → 命中旧缓存 → 展示 private 之前的数据。

### 修复

在 `invalidateCache` 中补充 Feed 缓存清理：
```java
// 删除 Feed 片段缓存
redis.delete("feed:item:" + id);

// 清空 Feed 页面本地缓存，强制下次从 DB 重建
feedPublicCache.invalidateAll();
```

### 面试话术

> "有一天发现文章设为 private 后前端还能看到。SQL 和权限校验都是对的，问题出在 Redis + Caffeine 两级缓存没有同步失效。可见性变更时只清理了详情页缓存，Feed 片段的缓存还在。修复后对涉及可见性变更的方法同步清理了所有缓存层。"
